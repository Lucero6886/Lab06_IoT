🔍 2. Tổng quan Lab

Tên chuẩn hóa: Triển khai Máy chủ Web nhúng (Embedded HTTP Server), Định tuyến URL (URL Routing) và Điều khiển Thiết bị qua Trình duyệt Mạng Cục bộ (WLAN).

Chủ đề kỹ thuật chính: Giao thức HTTP (Request/Response), Khái niệm Local IP, Phân giải Tên miền Đa hướng (mDNS), Kiến trúc Định tuyến URL (Endpoints), và Kỹ thuật nhúng mã nguồn Front-end (HTML/CSS) vào vùng nhớ Flash (PROGMEM) của Vi điều khiển.

Vị trí trong hệ thống: Tầng 6 (Web cục bộ và Tương tác Mạng LAN).

  • Sự chuyển dịch Mô hình (Paradigm Shift): Ở Lab 3 (Blynk), Lab 4 (MQTT) và Lab 5 (Telegram), thiết bị của bạn phụ thuộc hoàn toàn vào Máy chủ Đám mây (Cloud) ở nước ngoài. Ở Lab 6 này, ta giành lại quyền tự chủ: Chính bo mạch ESP8266 của bạn sẽ trở thành một Máy chủ Web (Web Server) độc lập. Dù đứt cáp quang biển (mất WAN), hệ thống vẫn hoạt động trơn tru trong mạng nội bộ (LAN) của tòa nhà.
  • Chuẩn bị cho tương lai: Hiểu tường tận vòng đời của một gói tin HTTP Request (Get/Post) và HTTP Response (Mã 200, 404) ở bài này là nền tảng bắt buộc để bạn giao tiếp với hệ quản trị CSDL đám mây (Firebase REST API) ở Lab 7.

💡 3. Bản chất Học thuật: Giành lại Quyền tự chủ (Local LAN)

ESP8266 Đóng Vai Trò Là Một Máy Chủ Web Độc Lập

Trong Lab này, vi điều khiển nhỏ bé của bạn sẽ vận hành một dịch vụ mạng hoạt động dựa trên giao thức phi trạng thái (Stateless Protocol) HTTP:

  1. Mạng Cục bộ (Local Area Network - LAN): Khi NodeMCU kết nối vào Router WiFi nhà bạn, Router (qua giao thức DHCP) sẽ cấp cho nó một địa chỉ IP nội bộ (Ví dụ: 192.168.1.15). Địa chỉ này bị ẩn sau bức tường lửa NAT. Không ai từ ngoài Internet có thể nhìn thấy nó. Chỉ Laptop/Điện thoại đang bắt chung cục WiFi đó mới có quyền giao tiếp cục bộ.
  2. Chu trình HTTP Request (Yêu cầu): Khi bạn mở Google Chrome trên Laptop, gõ http://192.168.1.15/led_on. Trình duyệt Chrome đóng vai trò là Client, nó mở một Socket TCP đến Cổng số 80 của ESP8266 và gửi đi một gói tin HTTP GET Request.
  3. URL Routing (Định tuyến API): Mạch ESP8266 nhận được gói tin, nó bóc tách lấy cái đuôi URL là /led_on. Hệ thống sẽ tra từ điển (Router) xem cái đuôi này được chỉ định chạy hàm C++ nào. Nhận diện đúng, nó kích hoạt hàm digitalWrite bật đèn.
  4. HTTP Response (Phản hồi): Xử lý xong phần cứng, ESP8266 BẮT BUỘC phải đóng gói một thông điệp trả về (Status 200 OK) đính kèm nội dung là các thẻ <html>. Chrome nhận được mã HTML này sẽ tự động vẽ (Render) ra giao diện.

🎯 4. Mục tiêu học tập

Sau khi hoàn thành Lab 6, sinh viên cần CHỨNG MINH ĐƯỢC các năng lực:

  1. Phân tích được vòng đời của một HTTP Request (GET) và HTTP Response theo chuẩn kiến trúc mạng máy tính.
  2. Thiết kế thuật toán Định tuyến (URL Routing): Sử dụng thư viện ESP8266WebServer để ánh xạ chuẩn xác các đuôi Endpoints (như /, /led_on, /led_off) tới các hàm Callback C++ tương ứng.
  3. Ứng dụng công nghệ mDNS (Multicast DNS): Khắc phục nhược điểm IP động bị thay đổi liên tục bằng cách phân giải tên miền cục bộ (VD: truy cập bằng http://nhathongminh.local thay vì nhớ dãy số IP).
  4. Tối ưu hóa bộ nhớ tĩnh: Trình bày được kỹ thuật sử dụng từ khóa PROGMEM (kết hợp với macro FPSTR) để nhúng một Website hoàn chỉnh (HTML/CSS) vào bộ nhớ Flash (ROM) thay vì ăn mòn RAM.
  5. Cấu trúc Điều hướng (Redirect): Hiểu và sử dụng được mã trạng thái HTTP 303 (See Other) để chuyển hướng trình duyệt, chống hiện tượng người dùng bị kẹt lại ở một URL điều khiển gây lặp lệnh.

📚 5. Kiến thức nền tảng cần nắm vững

  • Sự tiến hóa của Thư viện (ESP8266WebServer): Trong rất nhiều giáo trình cũ, người ta dùng thư viện thô sơ WiFiServer và bắt sinh viên phải dùng vòng lặp để tự cắt từng chữ cái tìm chữ "GET /led". Phương pháp này cực kỳ nguyên thủy, dễ lỗi và dễ làm treo mạch. Chuẩn mực hiện nay yêu cầu dùng ESP8266WebServer. Nó cung cấp sẵn hàm server.on("/duong-dan", ten_ham) để tự động hóa hoàn toàn việc phân tích cú pháp (Routing).
  • Mã Trạng Thái (HTTP Status Codes): Là ngôn ngữ giao tiếp giữa Server (Mạch) và Client (Trình duyệt).
    👉 200 OK: Server báo Request hợp lệ, kèm theo dữ liệu HTML để vẽ web.
    👉 303 See Other: Server báo đã xử lý lệnh xong, ra lệnh ép Trình duyệt hãy tải lại trang chủ ngay lập tức. Cực kỳ hữu ích để tránh lỗi kẹt URL lệnh.
    👉 404 Not Found: Trả về khi Trình duyệt gõ sai một cái đường dẫn rác không tồn tại trong cấu hình.
  • MIME Types & Tiếng Việt (Charset): Trong lệnh server.send(200, "text/html; charset=utf-8", web_content), cụm từ text/html báo cho Chrome biết đây là mã Web để vẽ giao diện. Đặc biệt, thuộc tính charset=utf-8 BẮT BUỘC phải có để trình duyệt không hiển thị lỗi font các ký tự Tiếng Việt có dấu.

🚀 10. Quy trình Thực hành Từng bước

1 Pha 1: Nạp Bộ Não Máy Chủ & Định Vị Tọa Độ Mạng

Mục đích: Kích hoạt dịch vụ Web Server trên Port 80 và truy xuất địa chỉ IP nội bộ của Mạch.

  1. Mở Arduino IDE. Copy toàn bộ Mã nguồn Chuẩn học thuật (Phần 13) dán vào.
  2. Thử thách Code: Kéo lên đầu File, TỰ TAY XÓA các đoạn rào chắn <...> và điền tên WiFi, Pass WiFi, và Tên miền mDNS của bạn.
  3. Cắm cáp, Chọn COM Port và nhấn Upload.
  4. QUAN TRỌNG: Ngay khi Upload xong, mở Serial Monitor (115200 baud). Đè nút RST trên mạch. Quan sát quá trình Boot mạng.
  5. Chờ Serial in ra dòng vinh quang: ĐỊA CHỈ IP LOCAL CỦA BẠN LÀ: 192.168.x.x. GHI CHÉP LẠI cái IP đó!.

2 Pha 2: Thử nghiệm Giao diện Trình Duyệt (Client Testing)

Mục đích: Kiểm tra năng lực Phục vụ Web của Mạch ESP8266.

  1. Rút Smartphone ra (Đảm bảo điện thoại ĐANG KẾT NỐI VÀO CÙNG WIFI VỚI MẠCH).
  2. Mở trình duyệt Google Chrome hoặc Safari. Trỏ tay vào thanh URL.
  3. Nhập dãy số IP mà bạn vừa ghi chép (VD: 192.168.43.50), hoặc gõ tên miền mDNS (VD: http://sv-tuananh.local). Nhấn nút Go.
  4. Phép thuật: Cục Chip bé nhỏ của bạn vừa trả về một Giao diện Web Bảng Điều Khiển xịn xò ngay trên điện thoại!

3 Pha 3: Nghiệm thu Giao thức API Cục bộ

Mục đích: Chứng minh hệ thống HTTP Route hoạt động chính xác.

  1. Test API Downlink (Điều khiển): Chạm tay vào nút "BẬT ĐÈN" trên Web. Quan sát: Trình duyệt chớp tải lại (Redirect 303), Chữ đổi thành "ĐANG SÁNG", và Đèn vật lý trên Breadboard SÁNG RỰC tức thì. Độ trễ Local LAN gần như bằng 0.
  2. Test API Uplink (Cảm biến): Áp tay bóp chặt cảm biến DHT11. Lưu ý: Đây là Web tĩnh, số liệu sẽ không tự nảy. Hãy vuốt màn hình kéo từ trên xuống (Reload/F5) để ép Chrome gửi một HTTP GET Request mới. Lúc này web mới cập nhật Nhiệt độ cao thực tế!
  3. Test 404 Cảnh sát Mạng: Lên thanh địa chỉ Chrome, gõ thêm đuôi rác: 192.168.x.x/hack_he_thong. Nhấn Go. Trình duyệt sẽ trả về dòng chữ thô: "Loi 404: Khong tim thay tai nguyen URL...". Routing hoạt động hoàn hảo.

⚠️ 11. Trạm Kiểm Tra Kiến Thức (Interactive Q&A)

Bảo vệ Tư duy Mạng Cục Bộ trước Mentor:

Sinh viên phải trả lời lưu loát các câu hỏi này trước khi được phép copy mã nguồn:

1. Nếu đứt cáp quang biển (mất mạng Internet toàn cầu), Web Server của bạn có bật được đèn không?
💡 Gợi ý tư duy: VẪN BẬT ĐƯỢC BÌNH THƯỜNG! Hệ thống này chỉ cần WiFi Router cục bộ làm trạm thu phát sóng nội bộ (LAN). Nó không cần ping ra ngoài thế giới. Khác biệt hoàn toàn với Blynk/Telegram (sẽ bị tê liệt nếu đứt mạng quốc tế).
2. Mã lệnh khởi tạo là ESP8266WebServer server(80);. Tại sao lại dùng Cổng số 80?
💡 Gợi ý tư duy: Cổng 80 là cổng tiêu chuẩn quốc tế của giao thức duyệt web HTTP (Không mã hóa). Nhờ đó, trình duyệt Chrome của bạn tự động biết cách ngầm chui vào cổng này mà bạn không cần phải gõ URL rườm rà như http://192.168.1.15:80.
3. Dòng code server.on("/led_on", handleLedOn); (Routing) có ý nghĩa gì?
💡 Gợi ý tư duy: Nó là một "Từ điển định tuyến". Nếu nó phát hiện Trình duyệt Chrome gửi một gói tin đòi truy cập vào cái đuôi URL /led_on, nó sẽ tự động kích hoạt Cò súng để gọi hàm C++ có tên handleLedOn() ra phục vụ xử lý bật đèn.
4. Tại sao lại cảnh báo việc mở Web Local là cực kỳ rủi ro về mặt Bảo mật nếu dùng ở quán Cafe?
💡 Gợi ý tư duy: Vì Local HTTP mặc định KHÔNG CÓ ĐĂNG NHẬP (No Auth). Bất kỳ thiết bị rác nào bắt chung mạng WiFi của quán Cafe đều có thể dùng tool quét dải IP, tìm ra IP của mạch bạn và thoải mái chọc lệnh `/led_on` để phá hoại thiết bị của bạn.

🛑 12. Chống Học Vẹt & Lỗ hổng Trải nghiệm Web

5 Dấu hiệu Tố Cáo Sinh viên chỉ biết bấm Copy-Paste ở Lab 6:
  • Gõ IP lên trình duyệt, mạch chạy báo lỗi Loading xoay mòng mòng không ra web. Sinh viên gào lên "Code của Thầy lỗi rồi". Căn bệnh: Điện thoại đang xài 4G hoặc bắt nhầm Wifi phòng bên cạnh, làm sao đòi chọc vào IP Local của mạng phòng này được!
  • Code báo lỗi Compile đỏ rực ở dòng in HTML. Căn bệnh: Lên mạng chép code cổ đại dùng client.println("<h1 class=\"btn\">");. Bị kẹt trong ma trận dấu nháy kép (Escape Character Hell \"). Hãy xem kỹ thuật Raw String Literal giải cứu ở Phần 13.
  • Nhiệt độ trên Web cứ đứng im 30 độ mãi dù đã áp tay nóng hổi vào mạch. Sinh viên kết luận "Cảm biến đơ!". Căn bệnh: Quên mất bản chất HTTP là giao thức tĩnh (Stateless). Server không tự ném Data về nếu Client không chủ động F5 xin thêm (Refresh).
  • Bấm nút bật đèn, đèn sáng. Lỡ tay vuốt màn hình tải lại trang, cái đèn tự nhiên lại bị tắt đi bật lại 1 lần nữa ngoài ý muốn. Căn bệnh: Sao chép code thiếu kiến trúc Điều hướng. Trình duyệt bị kẹt lại ở cái đường dẫn /led_on, mỗi lần F5 là nó lại gửi lại lệnh chọc đèn. (Đọc kỹ thuật HTTP 303 Redirect ở mã chuẩn).
Cảnh báo Học thuật: Tối ưu Lấy mẫu Cảm biến (Rate Limiting)

- Rất nhiều giáo trình cho phép gọi hàm dht.readTemperature() nằm trực tiếp bên trong nội hàm handleRoot() (hàm phục vụ vẽ giao diện Web).
➔ Hậu quả tàn khốc: Nếu một người dùng "trẻ trâu" liên tục bấm Refresh (F5) trang web 5 lần trong 1 giây, cảm biến DHT11 sẽ bị ép đọc 5 lần liên tiếp. Vi mạch bên trong DHT11 sẽ bị treo cứng (vì nó chỉ cho phép đọc tối đa 1 lần/2 giây) và trả về lỗi NaN.
➔ Giải pháp Kỹ sư: Đọc mã nguồn chuẩn bên dưới. Hàm đọc DHT11 đã được tách riêng xuống vòng lặp loop(), bị khóa giới hạn bằng millis() định kỳ 2 giây. Hàm handleRoot() chỉ việc bốc cái số liệu đã được lưu sẵn trong biến toàn cục đem đi vẽ Web cực kỳ tốc độ!

/* ======================================================================
   KHỐI 3: HÀM XỬ LÝ ĐỊNH TUYẾN URL (CONTROLLER HANDLERS)
   Mỗi hàm tương ứng với 1 đường dẫn URL mà Trình duyệt gửi tới
   ====================================================================== */

// 1. Hàm Cấu trúc Trang chủ (Route gốc "/"): Lấy Data, Vẽ View HTML
void handleRoot() {
  // Ép kiểu Float thành String. Bẫy lỗi phần cứng nếu sensor chết
  String tempStr = isnan(currentTemp) ? "Lỗi Đọc" : String(currentTemp, 1);
  String humStr = isnan(currentHum) ? "Lỗi Đọc" : String(currentHum, 1);

  // Múc khối HTML từ bộ nhớ Flash ra Ram bằng lệnh FPSTR chuẩn hóa
  String html = FPSTR(MAIN_page);
  
  // VIEW TEMPLATE ENGINE: Tìm và Thay thế các Biến đánh dấu (%) bằng Dữ liệu thực tế
  html.replace("%TEMP_MARKER%", tempStr);
  html.replace("%HUM_MARKER%", humStr);
  html.replace("%LED_STATE_MARKER%", ledStateStr);

  // GÓI HÀNG TRẢ VỀ: Bắn mã phản hồi HTTP 200 OK + Gắn nhãn MIME Type text/html + UTF-8 để chống lỗi tiếng Việt
  server.send(200, "text/html; charset=utf-8", html);
}

// 2. Endpoint: Kích hoạt Lệnh Bật Đèn (Route "/led_on")
void handleLedOn() {
  digitalWrite(LED_PIN, HIGH); // Can thiệp Vật lý
  ledStateStr = "ĐANG SÁNG";
  
  // KỸ THUẬT REDIRECT (CHỐNG KẸT URL): 
  // Bắn HTTP Code 303 (See Other). Ra lệnh ép Trình duyệt Chrome không được đứng lại ở link /led_on
  // mà phải tự động nhảy ngược ngay lập tức về trang chủ ("/") để tải lại giao diện mới.
  server.sendHeader("Location", "/"); 
  server.send(303);
}

// 3. Endpoint: Kích hoạt Lệnh Tắt Đèn (Route "/led_off")
void handleLedOff() {
  digitalWrite(LED_PIN, LOW); // Can thiệp Vật lý
  ledStateStr = "ĐANG TẮT";
  
  server.sendHeader("Location", "/"); 
  server.send(303);
}

// 4. Cảnh sát Bẫy Lỗi HTTP 404 (Khi gõ bậy đường dẫn rác)
void handleNotFound() {
  // Trả về MIME Type text/plain (Chỉ là Text thô ráp, không phải trang Web)
  server.send(404, "text/plain; charset=utf-8", "Lỗi 404: Không tìm thấy tài nguyên URL này trên Máy chủ ESP8266!");
}

/* ======================================================================
   KHỐI 4: KHỞI TẠO SETUP
   ====================================================================== */
void setup() {
  Serial.begin(115200);
  dht.begin();
  
  pinMode(LED_PIN, OUTPUT);
  digitalWrite(LED_PIN, LOW);

  // KẾT NỐI VÀO MẠNG LAN CỤC BỘ
  WiFi.mode(WIFI_STA);
  WiFi.begin(ssid, password);
  Serial.print("\n>> Đang xin cấp Visa vào mạng WiFi");
  while (WiFi.status() != WL_CONNECTED) {
    delay(500);
    Serial.print(".");
  }
  Serial.println("\n>> WiFi Connected.");
  Serial.print(">> ĐỊA CHỈ IP LOCAL CỦA BẠN LÀ: ");
  Serial.println(WiFi.localIP());

  // KÍCH HOẠT DỊCH VỤ TÊN MIỀN ẢO mDNS (Trải nghiệm Web mượt mà, không cần IP)
  if (MDNS.begin(mdns_name)) {
    Serial.printf(">> Tên miền mDNS đã sẵn sàng tại: http://%s.local\n", mdns_name);
  }

  // KHAI BÁO BẢN ĐỒ ĐỊNH TUYẾN URL (ROUTING API REGISTRY)
  // Gắn Luật: Nếu Trình duyệt gõ Route X -> Phân công gọi Hàm C++ Y ra tiếp khách
  server.on("/", handleRoot);          
  server.on("/led_on", handleLedOn);   
  server.on("/led_off", handleLedOff); 
  server.onNotFound(handleNotFound);   

  // Bật Cầu dao Nguồn của Web Server
  server.begin();
  Serial.println(">> HTTP Server đang Lắng nghe túc trực tại Cổng Port 80...");
}

/* ======================================================================
   KHỐI 5: VÒNG LẶP CHÍNH LOOP
   ====================================================================== */
void loop() {
  MDNS.update(); // Bơm nhịp tim duy trì sóng phát thanh tên miền mDNS
  
  // Rình rập, Lắng nghe và Vớt các Gói tin Request từ Trình duyệt ném tới
  server.handleClient();

  // THUẬT TOÁN ĐỌC CẢM BIẾN NON-BLOCKING CÁCH LY
  // Tách biệt việc đọc DHT11 khỏi Hàm Load Web. 
  // Chỉ đọc cập nhật biến toàn cục 2 giây/lần. Tránh việc user Spam F5 làm chết cảm biến.
  if (millis() - lastDHTRead > 2000) {
    currentTemp = dht.readTemperature();
    currentHum = dht.readHumidity();
    lastDHTRead = millis();
  }
}

🛠 15. Debug và Xử lý sự cố Giao thức HTTP Cấp cao

Hiện tượng Trình duyệt Báo Lỗi Căn nguyên Kỹ thuật Lõi (Root Cause) Hành động Chữa cháy (Troubleshooting)
Trình duyệt Chrome xoay mòng mòng mỏi mòn rồi văng lỗi xám xịt This site can’t be reached (ERR_CONNECTION_TIMED_OUT). Lỗi Phân Rã Định Tuyến Mạng (Routing Isolation). Mạch và Điện thoại không nằm chung 1 dải IP mạng. Hoặc Router WiFi Giảng đường/Quán Cafe đã kích hoạt Tường lửa "AP Client Isolation" chặn các thiết bị LAN nhìn thấy nhau. Ngắt mạng ngoài. Giải pháp Tối thượng: Lấy điện thoại cá nhân tự phát Hotspot 4G/5G. Ép cả cái Laptop và bo mạch NodeMCU cùng chui vào bắt sóng điểm Hotspot đó để tạo thành 1 mạng LAN khép kín.
Trang Web trông như một bãi rác lộn xộn (các thẻ <html> trần trụi hiện phơi bày), không hề có bảng màu hay nút bấm bo góc CSS. Lỗi Bắt sai Định dạng Gói (MIME Type Mismatch). Chrome không biết Data vừa nhận là web hay là văn bản Text thô. Do hàm C++ khai báo trả về Header sai chuẩn. Kiểm tra Hàm handleRoot() trong C++. Tìm dòng lệnh server.send(200, "text/html; charset=utf-8", html);. Đảm bảo chắc chắn tham số thứ 2 viết đúng là "text/html" (Chứ không phải "text/plain").
Bấm nút trên Web ngon lành, Giao diện Web tự tin báo chữ "ĐANG SÁNG", nhưng Bóng Đèn vật lý trên bàn chết im lìm không nhúc nhích. Bất đồng bộ Phần cứng (Hardware Mapping Error). Web Server đang xuất lệnh mở điện ra đúng chân GPIO lập trình, nhưng bạn cắm sai lỗ vật lý trên Breadboard. Mở đèn pin Điện thoại soi lại sát sàn sạt Breadboard. Code quy định #define LED_PIN 12 (Lỗ D6). Phải cắm chân dài của LED nối tiếp qua trở tọc đúng vào lỗ D6.
Gõ tên miền ảo mDNS (Vd: http://sv-tuananh.local) không tải được Web, phải gõ số IP mới tải được. Hệ điều hành máy tính của bạn thiếu Dịch vụ Phân giải Đa hướng (Bonjour/Zeroconf). Lỗi này cực kỳ phổ biến trên các máy tính Windows cũ, trong khi iPhone/Macbook thì ăn ngay. Nếu xài Windows, hãy tải và cài bộ Apple Bonjour Print Services. Hoặc đơn giản hóa vấn đề: Hãy tạm chấp nhận sử dụng dãy số IP cứng để tiến hành Nghiệm thu.

Thử thách Phòng Lab (Kỹ năng Bảo mật & Front-End):

Nhiệm vụ 1: Lỗ Hổng Mạng LAN - Triển Khai URL Token Authentication Như đã cảnh báo ở phần 9, một khi bạn online trong cùng WiFi, gã hàng xóm gõ IP của bạn kèm đuôi /led_on là đèn nhà bạn sáng! Hãy thiết lập một lớp bảo mật Token đơn giản.
=> Nhiệm vụ tác chiến: Bạn hãy chỉnh sửa C++ hàm handleLedOn(). Sử dụng hàm server.hasArg("token") và server.arg("token") để yêu cầu đường dẫn bật đèn phải chứa đúng một mã khóa (Ví dụ: /led_on?token=123). Nếu không có Token hoặc sai mã số 123, hãy thẳng tay quăng lại lỗi mã HTTP 403 (Forbidden) hoặc 401 (Unauthorized) từ chối phục vụ!
Nhiệm vụ 2: AJAX Update (Nâng cấp Cập Nhật Mượt Không Chớp Trang) Việc người dùng phải F5 tải lại toàn bộ khung trang Web để xem nhiệt độ là một Trải nghiệm (UX) cực kỳ hạ cấp. Một Full-stack Developer sẽ sử dụng công nghệ AJAX (Asynchronous JavaScript) hoặc hàm Fetch API tân tiến.
=> Nhiệm vụ Tự Học Mở Rộng:
- Tại Back-end C++: Bạn thiết lập thêm 1 cái Endpoint API mới (VD: server.on("/get_data", [](){ server.send(200, "text/plain", String(currentTemp)); });).
- Tại Front-end HTML: Bạn chèn thêm thẻ <script>, viết một hàm JavaScript dùng setInterval tự động gửi lệnh fetch('/get_data') ngầm dưới nền để lôi số liệu về và thay ruột Text dán đè lên cái thẻ Nhiệt độ mà không hề làm chớp màn hình trang web.

Việc cộng dồn chuỗi kiểu đó cực kỳ dễ văng lỗi vì kẹt Dấu Nháy Kép (Escape Character). Hãy sử dụng kiến trúc thanh tao R"=====( HTML Của tôi ở đây )=====". Kỹ thuật lập trình hiện đại yêu cầu sự phân ly vô trùng rạch ròi giữa Tầng Giao diện (HTML) và Tầng Điều khiển Lõi (C++).
  • Kiến thức Bảo mật Tiên quyết: Đừng bao giờ lầm tưởng IP là Mật khẩu. IP 192.168.x.x giống như Số Nhà. Một khi em mở Web Server ở nhà, bất cứ vị khách nào vào chơi xin pass WiFi cũng có thể tự do mở Chrome gõ IP để tắt bật đèn phòng ngủ của em.
